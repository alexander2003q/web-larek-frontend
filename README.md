Структура проекта:

- src/ — исходные файлы проекта

- src/types/model/WebLarekApi.ts - описание типов для API приложения

# Архитектура MVP

Основным связующим звеном между моделями и отображениями является **EventEmitter**.
Он нужен для регистрации событий и реакции на них.

Модели и отображения ничего не знают друг о друге.
Все происходит через **EventEmitter**.

## Схема взаимодействия

**Поток данных:**
BasketView -> 'ui:add-basket' -> EventEmitter -> BasketModel.addItem(id) -> 'ui:change-basket' -> EventEmitter -> BasketView.render()

# Ключевой базовый тип для отображений

Базовым типом для всех отображений является:

**`View<ViewData, ViewSettings>`**

---

## **Параметры конструктора:**

- **`element`** — ссылка на корневой DOM-элемент, в который будет происходить рендеринг.

---

## **Основной метод:**

- **`render()`** — рендерит компонент внутри корневого элемента.

---

## **Обобщённые параметры типа:**

### 1. **`ViewData`**

Данные, которые будут участвовать в рендеринге **и отображаться пользователю**.

### 2. **`ViewSettings`**

Вспомогательная информация, которая **не отображается пользователю**, но используется для рендеринга:

- Названия CSS-классов
- Селекторы DOM-элементов
- **Callback-функции** (обратные вызовы)
- Обработчики событий
- Другие служебные настройки

View для отображения списка однотипных элементов.

**ListViewData**

**ListViewSettings**

- **item** – View для отрисовки каждого элемента списка

View для отображения модального окна.

**ModalViewData**

- **isError** – флаг ошибки (опционально), применяет стиль для отображения сообщения об ошибке

**ModalViewSettings**

- **close** – селектор для кнопки закрытия модального окна
- **content** – селектор для области отображения основного контента
- **footer** – селектор для области футера (нижней части) модального окна
- **contentView** – View для рендеринга основного контента
- **actions** – массив HTML-элементов для размещения в футере модального окна
- **activeClass** – CSS-класс, добавляемый при открытии модального окна
- **errorClass** – CSS-класс для стилизации сообщений об ошибке
- **onOpen** – колбэк, вызываемый при открытии модального окна (опционально)
- **onClose** – колбэк, вызываемый при закрытии модального окна (опционально)

# Модели (Models) в архитектуре MVP

## **1. Модель каталога товаров (ICatalogModel)**

**ICatalogModel** — это модель, которая хранит данные о всех продуктах, получаемых с бэкенда.

### **Структура данных:**

Каждый товар представлен в виде объекта с полями:

- **id** — уникальный идентификатор товара
- **title** — название товара
- **description** — подробное описание товара
- **image** — ссылка на изображение товара
- **category** — категория товара
- **price** — цена товара (может быть null)

### **Методы работы:**

- **setItems(products)** — полностью обновляет каталог товаров, заменяя старые данные новыми при получении данных от API
- **getItem(id)** — принимает id товара и возвращает соответствующий объект для доступа к информации о конкретном продукте

### **События:**

Поскольку каталог является статичным со стороны пользователя и не предусматривает возможности его редактирования, модель генерирует только одно событие:

- **catalog:change** — происходит в момент обновления данных о товарах. В ответ на это событие интерфейс должен перерисовать каталог, отобразив актуальный список продуктов.

---

## **2. Модель корзины (IBasketModel)**

**IBasketModel** — это модель, которая хранит данные о продуктах, добавленных пользователем в корзину.

### **Особенности хранения:**

Модель не содержит полной информации о товарах, а только их идентификаторы в виде **Set**. Это позволяет:

- Эффективно проверять наличие товара в корзине
- Избегать дублирования товаров

### **Методы работы:**

- **addItem(id)** — добавляет ID товара в корзину
- **removeItem(id)** — удаляет ID товара из корзины

### **События:**

- **changeBasket** — срабатывает при любом изменении состава корзины и используется для обновления отображения
- **ui:basket-add** — команда от пользовательского интерфейса, которая триггерит добавление товара в модель
- **ui:basket-remove** — команда от пользовательского интерфейса, которая триггерит удаление товара из модели

---

## **3. Модель заказа (IOrderModel)**

**IOrderModel** отвечает за управление данными формы заказа — способом оплаты и адресом доставки.

### **Поток данных:**

1. Когда пользователь вводит информацию в форму, срабатывает событие **changeOrder**
2. Эти данные обновляются в модели через метод **setData(data)**
3. После подтверждения формы происходит событие **submitOrder**
4. Данные сохраняются для отправки на сервер
5. Открывается следующее модальное окно для ввода контактной информации

---

## **4. Модель контактов (IContactsModel)**

**IContactsModel** управляет данными контактной информации — email и телефоном.

### **Поток данных:**

1. При вводе пользователем данных в форму контактов генерируется событие **changeContacts**
2. Актуальные данные обновляются в модели через метод **setData(data)**
3. Когда пользователь подтверждает ввод контактной информации, происходит событие **submitContacts**
4. На бэкенд отправляются все данные пользователя по заказу
5. Завершается оформление заказа

---

## **5. Модель состояния приложения (AppState)**

**AppState** предоставляет интерфейс для управления модальными окнами приложения.

### **Функциональность:**

- **openModal(modal)** — открывает указанное модальное окно

### **Доступные модальные окна:**

- **modal:preview** — просмотр деталей товара
- **modal:basket** — корзина покупок
- **modal:order** — оформление заказа
- **modal:contacts** — ввод контактных данных
- **modal:success** — подтверждение успешного оформления

# API-слой приложения (IWebLarekAPI)

## Что это такое?

Интерфейс для общения с бэкенд-сервером. Отвечает за все запросы к серверу и получение данных.

## Основные функции

### 1. Получение товаров

- **getProducts()** – получает весь список товаров из каталога
- **getProduct(id)** – получает детальную информацию об одном товаре по его ID

### 2. Оформление заказа

- **createOrder(order)** – отправляет данные заказа на сервер и создаёт новый заказ

## Основные типы данных

### ApiListResponse<Type>

Формат ответа от сервера для списков:

- **total** – общее количество элементов
- **items** – массив элементов

### Product (Товар)

Данные о товаре, которые приходят с сервера:

- **id** – уникальный ID товара
- **title** – название товара
- **description** – описание товара
- **image** – ссылка на картинку товара
- **category** – категория товара
- **price** – цена (может быть null)

### ProductCategory (Категории товаров)

Все товары делятся на 5 категорий:

- софт-скил – программы и курсы
- хард-скил – оборудование и инструменты
- другое – разные товары
- дополнительное – аксессуары
- кнопка – кнопки и переключатели

### PaymentMethod (Способы оплаты)

Два варианта оплаты заказа:

- online – онлайн-оплата картой
- upon receipt – оплата при получении

### OrderData (Данные для заказа)

Что нужно отправить на сервер для оформления заказа:

- **payment** – способ оплаты
- **email** – email покупателя
- **phone** – телефон покупателя
- **address** – адрес доставки
- **total** – общая сумма заказа
- **items** – список ID товаров в заказе

### OrderResult (Результат заказа)

Что приходит в ответ при успешном оформлении:

- **id** – номер заказа
- **total** – подтверждённая сумма
